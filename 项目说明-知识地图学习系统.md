## 1. 项目缘起与目标

一位开发者父亲,有一对**龙凤胎**,2026 年 9 月上小学一年级。他想用 AI 辅助、陪两个孩子一起成长学习,并用短视频记录过程中"AI 以怎样的形式参与孩子学习"。

叙事框架是**师徒**:儿子是大师兄,女儿是小师妹,父亲是师傅。

**根本目标**(一切设计都服务于此):
- 让孩子**明确学习的目的**;
- **产生学习的兴趣**;
- 长出**自驱力**和**对知识的好奇心**。

整套系统的核心隐喻是:**好奇 → 探索 → 惊喜**。AI 是师傅手里的"暗器"(幕后加工孩子的问题、生成探险),不是孩子面前的"答案机"。秒答会填平好奇心赖以存在的"已知与未知之间的缝隙"。

## 2. 两个孩子(关键设计输入)

两个孩子性格、学习态度迥异,而且**强项与短板恰好镜像**——一个强在哪,另一个正好弱在哪。这是整套系统最大的设计资产。

| | 大师兄(儿子) | 小师妹(女儿) |
|---|---|---|
| 性格 | 温和、爱让妹妹;**看见妹妹有啥就也想要啥**(爱比较/跟随) | **有主见,不在乎别人怎么说**(自主性强) |
| 对学习 | 看电视能自学很多汉字、数字(**自带学习驱动力**);对绘画不感兴趣 | **知道难就不想尝试**(回避挑战);擅长跳舞、画画 |
| 已点亮的强项 | 敢学、爱琢磨 | 有主见、不怕评价 |
| 成长边(要补) | 弄清"我自己想要什么"(别总跟着妹妹) | 知难也敢"先试一下" |

**镜像 → 互为师傅。** 大师兄强在"敢试/爱学",正是小师妹要补的;小师妹强在"有自己的想法",正是大师兄要补的。设计上让他们**互相教**:哥哥示范"先试一下又不会怎样",妹妹示范"有自己的主意、不必跟随"。

**专属注意点:**
- 大师兄爱比较 → **绝不能做排行榜 / 两人 PvP**,否则会变成羞辱机器。各修各的路、各有各的树。他的"我也想要"要导进**自己技能树的解锁机制**(去点亮你自己的格子),而不是去要妹妹的。
- 小师妹怕难但不在乎评价 → 她怕的不是丢脸,是"自己一上手做不好"。解法是**降低第一次尝试的门槛**、把"做得烂"变成正常甚至好笑(例:"丑画博物馆",第一版越丑越光荣)。
- 两人的"独门绝技"天然分岔:大师兄偏文字/数字/逻辑,小师妹偏画画/跳舞/编故事。不重叠本身就在告诉爱比较的哥哥:"你有你的,她有她的,没法比。"

## 3. 核心概念框架

这几条是整个产品的"世界观",所有视图和功能都应与之自洽。

### 3.1 宏观学习地图:按"想要什么"组织,不按学科
让孩子**站得更高**,先有宏观、目标性的认知,再进细节(教育学上对应布鲁纳"螺旋式课程"、奥苏贝尔"先行组织者")。
关键设计:地图**按"你可能想要什么"组织,而不是按学科**。因为终极目标是"有一天孩子想到要什么,能找到那条路"——组织逻辑必须是 **欲望 → 知识**。若按"语文/数学/英语"画,会退化成课程表、把学校的分类提前塞给孩子。
v0 的几块"大陆"(故意只放几块,让它是地图不是教科书):**万物之理**(自然/数/规律)、**众生之间**(文字/语言/历史/人情)、**心之所向**(画/舞/乐/故事)、**巧手造物**(手作/搭建/代码/设计)、中心是**知己之道**(我是谁、想要什么)。
数学就是"万物之理"里的一片地,它自己内部又是一张小地图——这正是本项目当前聚焦的部分。

### 3.2 两层结构:地图 vs 技能树
- **地图 = 可能的疆域**:永远画不完、也不用走完,只负责让孩子知道"有这么多地方可以去"。用于俯瞰、惊奇、定位。
- **技能树 = 真正在走的路**:会生长。
- **桥**:孩子在地图某片反复停留、问个不停,那份好奇就"播种"成技能树上的一根新枝(欲望落在地图 → 长在树上)。路不是家长铺的,是好奇自己长出来的。

### 3.3 真身 vs 化形(数学的关键洞察)
数学痛苦,多因认知方法错——以为知识点是散乱的。其实**出题永远围绕知识点,只是换了形式**,且"换形式"有迹可循。
- **真身** = 知识点的深层结构;**化形** = 它穿上的各种题型外衣。
- 看懂真身,**没见过样子的新题也认得出它是谁**(认知科学:专家按"原理"分类题目,新手按"长得像不像"分类)。
- **出题者视角(problem posing)**:能给一个知识点自己出三道长得不一样的题,是最深的一档理解。放进师徒框架:高手不只是接招,是能给师弟师妹设擂台。这是个可玩、可拍、不靠低级奖励的任务。
- **刀刃**:方向必须是"先懂原理 → 再看它能变成哪些样子",**绝不能反过来"先背会十二种题型 → 套模板"**。后者就是制造"痛苦数学"的刷题套路。试金石:孩子遇到没见过的新题,是慌(套路派)还是去扒它的真身(原理派)。

### 3.4 先后顺序 = 前置依赖(技能树是 DAG,不是树)
"该先学什么"是**另一种关系**:不是"归属"(属于哪条脉),而是"前置依赖"。关键:**前置是跨脉的**。例:学"进位加法·凑十"必须先会"认识11–20"(数脉)**和**"10以内加减"(加减脉)。树里每个节点只能有一个父亲,画不出这种跨脉依赖——所以它本质是**有向无环图(DAG)**,即真正的游戏技能树:一个技能可有多个前置,前置全点亮才解锁。
附带洞察:这条依赖链大体顺着上册→下册走,**反过来印证了课本为什么这么排——不是老师的经验,是知识本身的依赖决定的**(接上"想从编者视角看为什么这么教")。

### 3.5 不同学龄阶段 = 同一张地图,不同关系
不是画好几张图,而是同一张地图、孩子与它的关系在变:
- 小(6–8):**惊奇**——知道这些大陆存在,到处看看,攒好奇心。
- 中(9–12):**偏好**——我好像更爱往这边走,自愿深挖。
- 大(13+):**方向**——我要的东西在地图哪一块,自己规划路线。
**家长纪律**:可以握着全图,但每次只展开孩子正走近的那片地,别一次铺开整块大陆——否则全图会变成期待清单、地图退回路线图。

## 4. 已做出的三个视图(原型现状)

三个原型**同源**:节点是同一批知识点,只是表达的关系/侧重不同。均为零依赖单文件 HTML、暗色"墨谱/夜林"风、字体 Noto Serif SC + IBM Plex Mono。

| 视图 | 文件 | 表达的关系 | 适合回答 | 决定 |
|---|---|---|---|---|
| **生命之树** | `grade1-math-tree-v2.html` | 归属(脉 → 知识点 → 题型),逐枝生长 | **整体有哪些**(全局概览) | **保留** |
| **修炼路线(技能树)** | `grade1-math-skilltree.html` | 前置依赖(DAG),点亮解锁 | **什么时候该学什么**(学习顺序) | **保留** |
| 真身谱(墨谱) | `grade1-math-atlas.html` | 按真身六脉重排,点开看化形;静态俯瞰 | 俯瞰真身/化形对应 | 早期草稿,**可选保留/归档**;数据已并入统一源 |

**生命之树**:左→右真正的树形。种子(一年级数学)在左,点节点向右长出主干→枝→题型花苞;每个节点带名字;悬停在左下读出真身。交互是"亲手让它长出来"。
**修炼路线**:横向七个境界(起手→筑基→小成→登堂→入室→大成→出师)为列,连线=前置且**按来源脉着色**(于是跨脉依赖一眼可见);一开始只有无前置的"起手"节点可学,点亮后续解锁;点已学会的会连带收回后续以保持一致;悬停读出真身、前置、还缺什么。

> 这三个文件可作为**参考实现**带入开发模式;但正式网站应改为"统一数据源 + 视图组件"的结构(见第 5、8 节),而不是各自硬编码一份数据。

## 5. 统一数据模型 ★

**这是整个网站的"单一真相源"。** 三个视图都只是这份数据的不同渲染。新增知识点 = 往数据里加一条;优化沉淀 = 改某条的字段。

### 5.1 schema
脉(vein)定义:

```json
{
  "shu": { "name": "数 · 十进制",    "color": "#d9b65a" },
  "jj":  { "name": "加与减",         "color": "#e07a55" },
  "bi":  { "name": "比较",           "color": "#6fb7ad" },
  "fl":  { "name": "分类与位置",     "color": "#8aa6d8" },
  "tu":  { "name": "图形",           "color": "#8fbf86" },
  "rmb": { "name": "人民币 · 元角分", "color": "#c98bb5" }
}
```

境界(tier,用于技能树分层):`["起手","筑基","小成","登堂","入室","大成","出师"]`

每个知识点(node)字段:

| 字段 | 说明 | 用于 |
|---|---|---|
| `id` | 唯一标识 | 全部(前置引用靠它) |
| `name` | 知识点名 | 全部 |
| `vein` | 所属脉(见上) | 树形(归属)、配色 |
| `tier` | 境界层级(0..6) | 技能树(分层) |
| `vol` | 上 / 下(册) | 标注 |
| `truth` | 真身一句话(深层结构) | 全部(悬停读出) |
| `forms` | 化形/常见题型(数组) | 树形叶子、出题者任务 |
| `req` | 前置知识点 id(数组,**可跨脉**) | 技能树(解锁/连线) |
| `from` | 缘起/叙事(它从哪长出来,文案) | 树形/文案,可选 |
| `note` | 编者视角(为什么这么教),可选 | 文案,可选 |

> 视图各取所需:**树形**用 `vein`(归属)+`forms`;**技能树**用 `tier`+`req`。两者读同一批记录。

### 5.3 两个视图的核心算法(要点)
- **生命之树(树形布局,左→右)**:按 `vein` 建三层树(脉→知识点→题型);列 `x = 深度 × 列宽`,叶子按行排,内部节点 `y = 子节点首尾的中点`(经典 dendrogram)。点击展开/收起,新节点从母节点位置插值"抽"出去 → 生长动画。
- **修炼路线(技能树/DAG)**:列 = `tier`,列内竖直居中堆叠;边 = 每个节点的 `req`(prereq → node),**按来源节点的脉色着色**,带箭头。状态机:`learned` 集合;`done`(已在集合)/`open`(req 全在集合)/`locked`。点 open → 学会;点 done → 连带取消所有(传递)依赖它的后续;locked → 只提示缺哪些前置。
- **避坑提醒(来自原型踩过的雷)**:SVG 里给 `fill/stroke` 用 `setAttribute` 时**不要写 `var(--x)`**(CSS 变量在表现属性里不生效会变黑),用字面色值;动态创建的节点 `<g>` **务必 append 到画布分组**里(否则只剩连线、没有节点和文字)。

## 8. 网站架构与迭代方式建议

父亲的元目标:**新想法能具像化,已实施的能优化沉淀**。架构应让"数据=真相,视图=渲染器,加一个知识点=改数据"。

**推荐结构(框架无关,先静态即可):**
```
/data
  curriculum.schema (字段定义,见 5.1)
  math-grade1.json   (见 5.2,可直接用)
  …future: chinese-grade1.json / english-grade1.json
/views
  TreeView      (生命之树:消费 vein + forms)
  SkillTreeView (修炼路线:消费 tier + req)
  …future: MapView(宏观"想要什么"地图,作为顶层导航)
/app
  顶层导航在多个视图间切换;主题(暗/儿童)可切换
```
- **技术选型(待定)**:当前原型是零依赖单文件 HTML,可直接托管,也是最好移植的种子。正式个人站可选 **Astro / Vite+React / Next** 等静态方案;数据存为 JSON;静态部署。v1 **无需后端**。
- **可扩展性**:schema 是通用的——把 math-grade1.json 换成语文、英语的同构数据,同一套视图即可复用。这就是从"一年级数学"长成"整张学习地图"的路径。
- **迭代工作流**:新想法 → 加数据 / 加一个视图组件;沉淀 → 打磨 `truth`/`forms`/`req`、给数据加版本。
- **未来功能候选**(按"护栏"第 9 节把关):
  - 出题者任务:让孩子给某知识点自己出一道"化形题";
  - "答对才开花":技能树解锁与孩子真实掌握挂钩(注意别滑成低级奖励);
  - 宏观"想要什么"地图作为顶层入口,数学只是其中一片大陆;
  - 短视频记录的轻量挂钩(不为出片设计环节、不在孩子面前强调"要发出去")。

## 9. 护栏:不能违背的原则

> 给将来在开发模式里加功能的人(或 AI agent)的硬约束。这些直接关系到"自驱力/好奇心"这个根本目标,违背了就南辕北辙。

1. **AI 是暗器,不是答案机。** 面向孩子时让它会反问、卖关子、陪做实验,而非秒给完整答案。
2. **当心过度理由效应。** **绝不给孩子已经热爱、本来就会自己做的事发积分/奖励**(例:大师兄的认字数数、小师妹的画画跳舞)——会把"我喜欢"换成"我为了分"。
3. **升级给称号和权利,不给数字。** 6 岁孩子对"经验值1250"无感;魔力在跨过门槛那一刻的仪式、新称号、新解锁的权利(自己选探险、教师傅一招、主持家庭会议)。
4. **不做排行榜 / 不做两人 PvP。** 尤其因为大师兄爱比较。各修各的路、各有各的树。要合作就设共同的门派目标,两人用不同方式贡献。
5. **数学:先懂原理→再看伪装,绝不"背题型套路"。**
6. **镜头不能偷换动机。** 不偷拍、不为出片设计环节、不在孩子面前强调"这段要发出去";隐私层面长期公开发布需父亲自行权衡。
7. **技能树是镜子,不是贿赂。** 它让孩子看见"我正在变成谁"(会观察的人、不怕难题的人),服务胜任感与身份认同。

## 10. 决策记录与待定问题

**已定:**
- 保留两个视图:生命之树(整体有哪些)+ 修炼路线(什么时候学什么);真身谱降为可选/归档。
- 数据驱动、单一真相源;前置用 `req` 字段表达(DAG)。
- 暗色"墨谱/夜林"风;脉色语义固定(第 7 节)。
- 先聚焦一年级数学,schema 通用以便扩展到其他学科。

**待定(进开发模式后定):**
- 技术栈:Astro / React / 纯静态?部署在哪?
- 是否做"儿童模式"(亮色大字)主题?
- 真身谱是否保留?
- `req` 前置关系含判断成分(如"求相差"挂在"比大小"+"退位减法"),需父亲逐条校正确认。
- 是否引入持久化(记录孩子已点亮的进度)?用本地存储还是后端?
- 短视频记录如何与系统挂钩(尽量轻量)。

## 11. 给开发模式的第一步任务

建议的最小可用第一版(MVP)落地顺序:

1. **建仓 + 数据**:新建项目,放入 `/data/math-grade1.json`(直接用第 5.2 的数据)和 schema 说明。
2. **两个视图组件**:`TreeView`(读 vein+forms,左→右生长树)与 `SkillTreeView`(读 tier+req,分层 DAG、点击解锁)。可参考三个原型 HTML 的实现与"避坑提醒"(第 5.3)。
3. **顶层导航**:一个页面,在两个视图间切换;沿用第 7 节设计 token。
4. **静态部署**:先上线一版能用、能迭代的骨架。
5. **回填校正**:父亲逐条核对 `req` 前置 与 `forms` 题型,沉淀进数据。

之后再按第 8 节的"未来功能候选",在第 9 节护栏内逐步加(出题者任务、答对开花、宏观地图、扩展到其他学科)。
